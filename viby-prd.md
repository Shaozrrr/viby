## 2. 产品概述

微步 Viby 是一个面向 Vibe Coding 创作者的极简作品社区。

用户可以上传自己通过 Cursor、Claude Code、Lovable、Bolt、v0、Replit 等 AI 编程工具完成的网站、应用、小工具、插件、游戏或实验项目，并添加作品描述、作品链接、GitHub 链接、技术栈、AI 工具标签等信息。

其他用户可以在社区中浏览、发现、访问、点赞和收藏这些作品。

微步要解决的核心问题是：

> AI 编程作品做出来之后，应该发到哪里、如何被看见、如何沉淀为个人作品集。

## 3. 产品定位

### 3.1 一句话定位

微步是中文 Vibe Coding 创作者的极简作品社区。

### 3.2 产品主张

让每一个用 AI 做出来的小作品，都有一个可以被看见、被访问、被收藏的展示页。

### 3.3 产品边界

第一版不做以下方向：

- 不做在线 IDE
- 不做代码托管
- 不做复杂论坛
- 不做私信社交
- 不做项目协作
- 不做模板市场
- 不做付费课程平台

微步第一版只聚焦一件事：

> 让 Vibe Coding 作品被发布、展示、发现。

## 4. 背景与机会

AI 编程工具正在快速普及，越来越多人可以通过自然语言完成可访问的网站、工具和应用。

但这些作品目前缺少合适的展示场景：

- GitHub 偏代码，不适合普通浏览者理解作品价值。
- Product Hunt 偏成熟产品发布，门槛较高。
- 社交媒体生命周期短，内容很快沉没。
- 微信群、朋友圈传播范围有限，难以持续被发现。
- 个人网站搭建成本高，缺少社区流量。

因此，市场存在一个空位：

> 一个低门槛、轻量化、面向 AI 编程作品的展示社区。

## 5. 目标用户

### 5.1 核心用户

#### AI 编程新手

- 刚开始使用 Cursor、Claude Code、Lovable 等工具。
- 做出了第一个小网站、小工具或 Demo。
- 希望被看见、获得正反馈和成就感。

#### AI Builder / 独立开发者

- 持续通过 AI 工具构建作品。
- 需要一个集中展示作品的主页。
- 希望增加作品曝光、GitHub 访问和个人影响力。

#### 浏览者 / 学习者

- 想看别人用 AI 做出了什么。
- 想找灵感、技术栈、项目参考。
- 可能未来转化为创作者。

### 5.2 早期重点用户

第一阶段优先服务：

- 中文 Vibe Coding 用户
- Cursor / Claude Code 用户
- 独立开发者
- AI 工具爱好者
- 正在学习编程或 AI 编程的新手

## 6. 用户痛点

### 6.1 作品无处展示

很多作品只是小工具、小网页或实验项目，不适合上 Product Hunt，也不适合只放 GitHub。

### 6.2 作品难以长期被发现

作品发在社交媒体后，通常只有短期曝光，几天后几乎无法再被找到。

### 6.3 创作者缺少作品集沉淀

创作者做了很多小作品，但缺少一个统一主页来展示自己的 AI 编程能力。

### 6.4 浏览者缺少灵感来源

想学习 Vibe Coding 的人，很难系统地看到别人做了哪些真实作品。

### 6.5 作品可信度不足

很多作品只有截图或宣传语，缺少 GitHub、技术栈、AI 工具等可验证信息。

## 7. 解决方案

微步通过以下方式解决上述问题：

- 提供极简发布表单，让用户快速提交作品。
- 为每个作品生成独立详情页。
- 支持作品访问链接、GitHub 链接、AI 工具、技术栈等信息。
- 提供首页作品流、标签筛选、精选推荐。
- 提供用户主页，沉淀个人作品集。
- 通过点赞、收藏、精选等轻互动机制促进发现。
- 通过审核、举报和后台管理保障内容质量。

## 8. 竞品分析

### 8.1 Product Hunt

优势：

- 发布机制成熟。
- 有投票、评论、榜单。
- 适合成熟产品曝光。

不足：

- 发布门槛较高。
- 更适合英文市场。
- 生命周期集中在发布当天。
- 不适合半成品、小实验、小工具。

微步机会：

- 更低门槛。
- 更适合日常发布。
- 更适合中文 AI 编程作品。

### 8.2 GitHub

优势：

- 开发者信任度高。
- 代码和提交记录可验证。
- 适合开源协作。

不足：

- 展示体验弱。
- 普通用户难理解项目价值。
- 更偏代码，不偏作品浏览。

微步机会：

- GitHub 负责代码可信度。
- 微步负责作品展示和社区发现。

### 8.3 Dribbble / Behance

优势：

- 视觉展示强。
- 适合建立作品集。
- 创作者身份感强。

不足：

- 更偏设计作品。
- 不适合工具、应用、代码项目。
- 缺少技术栈和 GitHub 信息。

微步机会：

- 成为 AI 编程作品版的轻量展示社区。

### 8.4 Indie Hackers / Show HN

优势：

- 开发者和创业者质量高。
- 反馈相对深入。
- 适合讲产品过程。

不足：

- 门槛较高。
- 偏英文和资深开发者文化。
- 对新手不够友好。

微步机会：

- 更轻松。
- 更中文。
- 更适合新手展示小作品。

### 8.5 Lovable / Bolt / Replit 社区

优势：

- 与创作工具结合紧密。
- 作品和模板转化路径短。

不足：

- 绑定单一工具平台。
- 中立性不足。
- 不适合展示跨工具创作过程。

微步机会：

- 做跨工具的 Vibe Coding 作品层。

## 9. 产品护城河

### 9.1 中文 Vibe Coding 社区心智

抢占“中文 AI 编程作品社区”的早期定位。

### 9.2 结构化作品数据

沉淀作品类型、AI 工具、技术栈、GitHub、作者、热度等数据。

### 9.3 创作者主页资产

让微步成为用户的 AI 编程作品集，提高长期留存和迁移成本。

### 9.4 精选与发现能力

通过人工精选、标签、榜单和专题，让好作品持续被发现。

### 9.5 跨工具中立平台

不绑定任何单一 AI 编程工具，兼容 Cursor、Claude Code、Lovable、Bolt、v0、Replit 等生态。

## 10. V1.0 MVP 范围

V1.0 直接覆盖 MVP，目标是形成最小社区闭环：

> 发布作品 -> 浏览作品 -> 查看详情 -> 访问作品 / GitHub -> 点赞收藏 -> 查看作者主页。

### 10.1 首页

目标：让用户快速理解产品，并发现作品。

功能需求：

- 展示品牌名称、定位文案和发布入口。
- 展示最新作品流。
- 展示精选作品。
- 支持按标签筛选作品。
- 支持进入作品详情页。
- 支持进入用户主页。

作品卡片字段：

- 封面图
- 作品名称
- 一句话简介
- 作者昵称
- 标签
- AI 工具
- 点赞数
- GitHub 标识
- 发布时间

### 10.2 发布作品

目标：让用户低成本提交作品。

发布前置条件：

- 用户必须登录。

必填字段：

- 作品名称
- 一句话简介
- 作品访问链接
- 作品分类
- 作者信息

选填字段：

- GitHub 链接
- 封面图
- 详细描述
- 使用的 AI 工具
- 技术栈
- 构建时间
- 标签

校验规则：

- 作品名称不能为空。
- 作品访问链接必须为合法 URL。
- GitHub 链接如填写，必须为合法 GitHub URL。
- 一句话简介建议不超过 80 字。
- 详细描述建议不超过 1000 字。
- 每个作品至少选择 1 个分类。

发布状态：

- 第一阶段可采用先发布后审核。
- 被举报或后台隐藏的作品不在前台展示。

### 10.3 作品详情页

目标：让访客完整理解作品，并产生访问行为。

页面内容：

- 作品封面
- 作品名称
- 一句话简介
- 详细描述
- 访问作品按钮
- GitHub 按钮
- 作者信息
- AI 工具
- 技术栈
- 分类与标签
- 发布时间
- 点赞按钮
- 收藏按钮
- 举报入口

关键行为：

- 点击访问作品，跳转外部链接。
- 点击 GitHub，跳转 GitHub 仓库。
- 点击作者，进入用户主页。
- 点击标签，进入对应标签作品列表。

### 10.4 用户主页

目标：沉淀创作者身份和作品集。

页面内容：

- 头像
- 昵称
- 简介
- GitHub 链接
- X / 即刻 / 个人网站链接
- 发布作品列表
- 获得点赞总数
- 加入时间

### 10.5 登录注册

V1.0 建议支持：

- GitHub 登录
- 邮箱登录，可后置
- 用户资料编辑

优先推荐 GitHub 登录，因为它符合开发者和 AI Builder 用户习惯，也有助于后续做 GitHub 验证。

### 10.6 互动功能

V1.0 只做轻互动：

- 点赞作品
- 收藏作品
- 举报作品

暂不做：

- 评论
- 私信
- 关注
- 动态流

原因：

- 降低开发成本。
- 降低社区治理压力。
- 保持产品极简。

### 10.7 后台管理

后台用于保障内容质量。

功能需求：

- 查看作品列表
- 搜索作品
- 隐藏作品
- 恢复作品
- 设置精选
- 处理举报
- 管理分类和标签
- 查看用户列表

## 12. 用户流程

### 12.1 发布作品流程

```text
用户登录
-> 点击发布作品
-> 填写作品信息
-> 上传封面
-> 添加作品链接 / GitHub / AI 工具 / 技术栈
-> 提交作品
-> 作品上线
-> 作品出现在首页最新列表和用户主页
```

### 12.2 浏览作品流程

```text
访客进入首页
-> 浏览作品卡片
-> 点击作品详情
-> 查看作品描述
-> 点击访问作品或 GitHub
-> 点赞 / 收藏
-> 进入作者主页查看更多作品
```

### 12.3 举报处理流程

```text
用户点击举报
-> 填写举报原因
-> 后台生成举报记录
-> 管理员审核
-> 隐藏作品或保留作品
```

## 13. 权限规则

### 13.1 未登录用户

- 可浏览首页。
- 可查看作品详情。
- 可访问作品外链。
- 可访问 GitHub 链接。
- 不可发布作品。
- 不可点赞、收藏、举报。

### 13.2 登录用户

- 可发布作品。
- 可编辑自己的作品。
- 可删除自己的作品。
- 可点赞、收藏、举报。
- 可编辑个人主页。

### 13.3 管理员

- 可管理所有作品。
- 可隐藏或恢复作品。
- 可设置精选。
- 可处理举报。
- 可管理标签和分类。

## 14. 核心指标

### 14.1 内容指标

- 总作品数
- 每周新增作品数
- 含 GitHub 链接作品占比
- 含封面图作品占比
- 精选作品数量

### 14.2 用户指标

- 注册用户数
- 发布过作品的用户数
- 每周新增创作者
- 用户主页访问量
- 创作者留存率

### 14.3 行为指标

- 作品详情页访问量
- 作品外链点击率
- GitHub 点击率
- 点赞数
- 收藏数
- 标签点击率

### 14.4 MVP 成功标准

V1.0 上线后 30 天目标：

- 收录 200 个作品。
- 获得 100 个注册用户。
- 至少 50 个用户发布作品。
- 每周新增 30 个作品。
- 20% 以上作品包含 GitHub 链接。
- 作品详情页到外链点击率达到 20% 以上。

## 15. 冷启动策略

### 15.1 内容冷启动

- 人工收集 50-100 个优质 Vibe Coding 作品。
- 邀请作者认领作品。
- 做“精选作品”填充首页。
- 保证用户第一次进入时能看到足够多的优质内容。

### 15.2 用户冷启动

- 邀请 Cursor、Claude Code、Lovable、Bolt 用户入驻。
- 在即刻、X、小红书、微信群、GitHub 社区传播。
- 给早期创作者提供“首批创作者”标识。
- 鼓励用户分享自己的微步主页。

### 15.3 内容传播

- 每周发布一次“本周 Vibe Coding 作品精选”。
- 按工具做专题，例如“10 个 Cursor 做出来的小工具”。
- 按场景做专题，例如“适合新手参考的 AI 编程作品”。

## 17. 盈利模式规划

### 17.1 短期：不急于商业化

V1.0 阶段核心目标是内容和用户，不建议过早收费。

优先目标：

- 积累作品。
- 积累创作者。
- 建立品牌心智。
- 验证用户是否愿意持续发布。

### 17.2 中期：创作者增值服务

适合在用户主页和作品集价值成立后推出。

可收费功能：

- Pro 创作者主页
- 自定义主页样式
- 自定义域名
- 更多作品置顶位
- 数据分析
- 外链点击统计
- 高级作品集模板

目标用户：

- 独立开发者
- 求职者
- AI Builder
- 自媒体创作者
- 接外包的开发者

### 17.3 中期：作品推广

适合在平台有稳定浏览量后推出。

可收费功能：

- 首页推荐位
- 分类页置顶
- 每周精选赞助位
- 新品推广
- AI 工具专题赞助

注意事项：

- 必须清楚标注推广内容。
- 不能影响普通作品的自然曝光。
- 不能破坏社区信任。

### 17.4 中后期：招聘与项目机会

当平台积累足够多 AI Builder 后，可以连接需求方和创作者。

可盈利方向：

- 企业发布 AI Builder 招聘岗位
- 企业发布外包需求
- 创作者接单线索
- 项目需求撮合
- 高级人才搜索

目标客户：

- 初创公司
- AI 产品团队
- 外包需求方
- 需要快速做原型的企业

### 17.5 长期：模板与源码市场

当作品和作者足够多后，可开放交易型功能。

可盈利方向：

- 模板售卖
- 源码售卖
- AI 应用 Starter Kit
- 付费复刻教程
- 平台抽佣

前提条件：

- 作品质量稳定。
- 有支付和交付能力。
- 有版权和售后规则。
- 有基础审核机制。

## 18. 商业化优先级

推荐顺序：

1. 先做社区和内容。
2. 再做创作者 Pro 主页。
3. 再做推广位。
4. 再做招聘和项目撮合。
5. 最后做模板和源码交易。

不建议第一版直接做交易，因为会增加信任、支付、版权、售后和审核复杂度。

## 19. 风险与应对

### 19.1 内容质量不稳定

风险：用户上传低质量、重复或无效作品。

应对：

- 首页突出精选。
- 设置举报机制。
- 后台可隐藏作品。
- 通过冷启动人工引入优质内容。

### 19.2 作品链接失效

风险：很多 Demo 部署链接后续失效。

应对：

- 增加作品状态检测。
- 允许用户标记作品状态。
- 后续展示“可访问 / 已失效”。

### 19.3 垃圾广告

风险：有人提交广告、灰产或无关链接。

应对：

- 登录后发布。
- 新用户发布频率限制。
- 举报和后台审核。
- 黑名单机制。

### 19.4 社区冷启动困难

风险：用户只看不发，作品数量不足。

应对：

- 人工收录第一批作品。
- 邀请创作者入驻。
- 提供漂亮个人主页。
- 做每周精选内容传播。

### 19.5 定位被稀释

风险：后续功能过多，变成普通社区。

应对：

- 始终围绕 Vibe Coding 作品。
- 首页只展示作品，不做泛内容流。
- 用户主页服务作品集，而不是社交动态。

## 20. V1.0 验收标准

V1.0 上线时必须满足：

- 用户可以注册 / 登录。
- 用户可以发布作品。
- 用户可以编辑和删除自己的作品。
- 首页可以展示作品列表。
- 作品详情页可以正常访问。
- 作品可以跳转外部访问链接。
- 作品可以跳转 GitHub 链接。
- 用户主页可以展示个人资料和作品列表。
- 登录用户可以点赞、收藏、举报。
- 管理员可以隐藏作品、设置精选、处理举报。
- 移动端和桌面端均可正常浏览。

## 21. 产品原则

- 极简优先。
- 发布门槛低。
- 作品展示优先于社交。
- 创作者身份沉淀优先。
- 先做内容密度，再做复杂互动。
- 先做社区信任，再做商业化。
- 保持跨 AI 工具中立。

## 22. 核心结论

微步 Viby 的第一版应专注完成一个最小但完整的社区闭环：

> 用户发布 Vibe Coding 作品，其他人发现、访问、点赞、收藏，创作者通过主页沉淀自己的 AI 作品集。

V1.0 不追求复杂社交和商业化，而是优先建立内容密度、创作者心智和作品展示体验。后续在社区内容稳定后，再逐步扩展搜索、榜单、GitHub 验证、创作者 Pro 主页、推广位、招聘撮合和模板交易等商业化能力。
